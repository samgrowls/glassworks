# LLM Query Integration Design

**Version:** 1.0.0
**Date:** March 23, 2026
**Status:** Design Specification - Ready for Implementation
**Author:** Glassworks Security Team

---

## Executive Summary

This document specifies the architecture for **LLM-powered query interface** for campaign reports. Users can ask natural language questions about scan results, findings, and campaign metrics, with responses generated by NVIDIA's LLM API using context from campaign data.

### Key Capabilities

| Capability | Description |
|------------|-------------|
| **Package queries** | "Why was express@4.19.2 flagged?" |
| **Campaign queries** | "What's the detection rate?" |
| **Finding queries** | "Show me all GlassWare patterns found" |
| **Comparative queries** | "How does this compare to wave 5?" |

### Recommended Approach

**Option C: Both CLI and TUI** - Start with CLI single-query command (Phase 1), then add interactive CLI session (Phase 2), and finally TUI chat interface (Phase 3).

---

## 1. Use Cases

### 1.1 Package-Specific Queries

Users want to understand why specific packages were flagged.

**Example Queries:**
```
"Why was express@4.19.2 flagged?"
"What patterns were detected in lodash@4.17.21?"
"Show me the LLM reasoning for react-native-country-select"
"Is the detection of @iflow-mcp a false positive?"
```

**Expected Response:**
- Summary of findings for the package
- LLM reasoning from analysis
- Relevant code snippets
- Severity explanation

### 1.2 Campaign Metrics Queries

Users want to understand overall campaign results.

**Example Queries:**
```
"What's the detection rate for this campaign?"
"How many packages were scanned in wave 6a?"
"Which wave had the most malicious packages?"
"Show me the timeline of package publications"
```

**Expected Response:**
- Computed metrics from campaign data
- Tables or formatted statistics
- Trend analysis if comparative

### 1.3 Finding Pattern Queries

Users want to explore detection patterns.

**Example Queries:**
```
"Show me all GlassWare patterns found"
"What RDD attacks were detected?"
"List all packages with blockchain C2 indicators"
"Which packages have Unicode steganography?"
```

**Expected Response:**
- Filtered list of findings by category
- Package names with specific patterns
- Code examples or line references

### 1.4 Comparative Queries

Users want to compare across campaigns or waves.

**Example Queries:**
```
"How does this compare to wave 5?"
"Are there more false positives than last campaign?"
"Show me the trend across all waves"
"Which campaign had the highest detection rate?"
```

**Expected Response:**
- Cross-campaign comparison
- Historical trend data
- Statistical analysis

## 2. Architecture Options

### 2.1 Option A: CLI Command

**Interface:**
```bash
glassware-orchestrator campaign query <case-id> "question"
```

**Example Usage:**
```bash
# Single query
glassware-orchestrator campaign query wave-6-calibration "Why was express@4.19.2 flagged?"

# With output file
glassware-orchestrator campaign query wave-6-calibration "Show me all findings" -o findings.md

# With specific wave context
glassware-orchestrator campaign query wave-6-calibration --wave wave_6a "What was detected?"
```

**Pros:**
- Simple to implement
- Scriptable for automation
- Fits existing CLI paradigm
- Easy to pipe to other tools

**Cons:**
- No conversation history
- Must re-specify context each time
- Less interactive experience

**Implementation Effort:** 4-6 hours

---

### 2.2 Option B: TUI Query Interface

**Interface:**
```bash
glassware-orchestrator campaign monitor <case-id>
# Then press 'q' for query mode
```

**Example UI:**

**Features:**

- Natural conversation flow
- Context retained across queries
- Rich visual presentation
- Query history navigation

**Cons:**
- More complex implementation
- Requires TUI framework (ratatui)

### 2.3 Option C: Both (Recommended)

**Phased Implementation:**

| Phase | Feature | Effort | Priority |
|-------|---------|--------|----------|
| **Phase 1** | CLI single query | 4-6 hours | High |
| **Phase 2** | Interactive CLI session | 6-8 hours | Medium |
| **Phase 3** | TUI query interface | 2-3 days | Low |

**Rationale:**
- Phase 1 delivers immediate value with minimal effort
- Phase 2 adds conversational context for power users
- Phase 3 provides polished UX for regular users

## 4. Context Building

### 4.1 Data to Include in Prompt

**Essential Context (Always Included):**

```rust
pub struct QueryContext {
    /// Campaign metadata
    pub campaign: CampaignMetadata,

    /// Summary statistics
    pub summary: CampaignSummary,

    /// Wave results
    pub waves: Vec<WaveSummary>,

    /// All findings (or filtered subset)
    pub findings: Vec<FindingSummary>,

    /// LLM analysis results (if available)
    pub llm_analyses: Vec<LlmAnalysis>,

    /// User's specific query
    pub query: String,
}
```

**Context Data Details:**

| Data Type | Content | Token Estimate |
|-----------|---------|----------------|
| Campaign metadata | Name, case ID, dates, settings | ~100 tokens |
| Summary stats | Scanned, flagged, malicious counts | ~50 tokens |
| Wave results | Per-wave tables | ~200-500 tokens |
| Findings | Category, package, severity, description | ~50-200 per finding |
| LLM analyses | Verdict, confidence, reasoning | ~100-300 per analysis |
| Package details | Name, version, files, code snippets | ~200-1000 per package |

### 4.2 Handling Large Reports (Token Limits)

**Challenge:** Campaign reports can exceed LLM context windows.

**NVIDIA API Context Limits:**

| Model | Context Window | Use Case |
|-------|---------------|----------|
| `qwen/qwen3.5-397b-a17b` | 256K tokens | Full campaign context |
| `meta/llama-3.3-70b-instruct` | 128K tokens | Standard queries |
| `moonshotai/kimi-k2.5` | 256K tokens | Large context |

**Strategy:**

1. **Tiered Context Loading**
   - Always include: campaign metadata + summary
   - Include if query mentions: specific wave/package/finding
   - Include if tokens remain: related findings

2. **Smart Filtering**
   ```rust
   // Filter findings based on query intent
   fn filter_findings(findings: &[Finding], query: &str) -> Vec<Finding> {
       let query_lower = query.to_lowercase();

       // Package-specific query
       if let Some(pkg) = extract_package_name(&query_lower) {
           return findings.iter()
               .filter(|f| f.package.contains(&pkg))
               .cloned()
               .collect();
       }

       // Category-specific query
       if let Some(category) = extract_category(&query_lower) {
           return findings.iter()
               .filter(|f| f.category == category)
               .cloned()
               .collect();
       }

       // Default: return top findings by severity
       findings.iter()
           .sorted_by_key(|f| f.severity)
           .take(50)
           .cloned()
           .collect()
   }
   ```

3. **Chunking Strategy**
   ```rust
   pub struct ChunkedContext {
       /// Always included
       pub core: CoreContext,

       /// Included if relevant
       pub relevant: Vec<ContextChunk>,

       /// Available on follow-up
       pub available: Vec<ContextReference>,
   }

   pub enum ContextChunk {
       WaveResults { wave_id: String, data: String },
       PackageFindings { package: String, data: String },
       CategoryFindings { category: String, data: String },
       LlmAnalysis { package: String, data: String },
   }
   ```

### 5.2 New QueryAnalyzer

**Recommended:** Create a new `QueryAnalyzer` specifically for queries.

```rust
//! Query Analyzer Module
//!
//! Provides LLM-based query answering for campaign reports.

use crate::campaign::query::context::QueryContext;
use crate::llm::config::LlmConfig;
use crate::llm::rate_limiter::RateLimiter;

pub struct QueryAnalyzer {
    config: LlmConfig,
    client: reqwest::Client,  // Async client
    rate_limiter: RateLimiter,
}

impl QueryAnalyzer {
    pub fn new(config: LlmConfig) -> Self {
        let client = reqwest::Client::builder()
            .timeout(Duration::from_secs(120))
            .build()
            .unwrap_or_else(|_| reqwest::Client::new());

        let rate_limiter = RateLimiter::from_env();

        Self { config, client, rate_limiter }
    }

    /// Answer a query with campaign context
    pub async fn answer(
        &self,
        context: &QueryContext,
        query: &str,
    ) -> Result<QueryResponse, QueryError> {
        let prompt = self.build_prompt(context, query);

        // Estimate tokens
        let estimated_tokens = (prompt.len() / 4) as u32;

        // Acquire rate limit permission
        self.rate_limiter.acquire(estimated_tokens).await;

        // Build request with streaming support
        let request_body = serde_json::json!({
            "model": self.config.model,
            "messages": [
                {"role": "system", "content": QUERY_SYSTEM_PROMPT},
                {"role": "user", "content": prompt}
            ],
            "temperature": 0.3,  // Slightly higher for more natural responses
            "stream": true  // Enable streaming
        });

        // Make API call with streaming
        let response = self
            .client
            .post(format!("{}/chat/completions", self.config.base_url))
            .header("Authorization", format!("Bearer {}", self.config.api_key))
            .header("Content-Type", "application/json")
            .json(&request_body)
            .send()
            .await?;

        // Parse streaming response
        self.parse_streaming_response(response).await
    }

    /// Answer with streaming callback
    pub async fn answer_streaming(
        &self,
        context: &QueryContext,
        query: &str,
        callback: impl Fn(String) -> (),
    ) -> Result<QueryResponse, QueryError> {
        // Similar to answer() but streams chunks to callback
    }

    fn build_prompt(&self, context: &QueryContext, query: &str) -> String {
        // Use template from Section 4.3
    }
}

pub struct QueryResponse {
    pub answer: String,
    pub model: String,
    pub tokens_used: u32,
    pub latency_ms: u64,
}
```

### 6.3 Export Q&A to Report Appendix

**Export Format:**
```markdown
## Appendix B: Q&A Session

**Session Date:** 2026-03-23 14:30:00 UTC
**Case ID:** wave-6-calibration
**Total Queries:** 5

---

### Query 1
**User:** Why was express@4.19.2 flagged?

**Assistant:** express@4.19.2 was flagged due to...

---

### Query 2
**User:** What about lodash?

**Assistant:** ...
```

**Implementation:**
```rust
pub fn export_conversation(
    history: &ConversationHistory,
    output_path: &Path,
) -> Result<(), ReportError> {
    let mut report = String::new();

    report.push_str("## Appendix: Q&A Session\n\n");
    report.push_str(&format!("**Session Date:** {}\n", Utc::now().format("%Y-%m-%d %H:%M:%S UTC")));
    report.push_str(&format!("**Case ID:** {}\n", history.case_id));
    report.push_str(&format!("**Total Queries:** {}\n\n", history.messages.len() / 2));
    report.push_str("---\n\n");

    for (i, msg) in history.messages.chunks(2).enumerate() {
        report.push_str(&format!("### Query {}\n", i + 1));
        report.push_str(&format!("**User:** {}\n\n", msg[0].content));
        report.push_str(&format!("**Assistant:** {}\n\n", msg[1].content));
        report.push_str("---\n\n");
    }

    fs::write(output_path, report)?;
    Ok(())
}
```

```markdown
## Analysis for express@4.19.2

**Status:** Flagged (False Positive Likely)

### Findings

| Rule ID | Severity | Category | Line | Description |
|---------|----------|----------|------|-------------|
| GW003   | Medium   | invisible_character | 42 | Zero-width character detected |
| GW007   | Low      | rc4_pattern | 156 | RC4-like XOR pattern |

### LLM Assessment

The LLM analysis determined these are **likely false positives**:

1. **Zero-width character (GW003):** The character appears in a comment string, not executable code. This is common in internationalized comments.

2. **XOR pattern (GW007):** The pattern is used for simple data encoding, not encryption. It's a common obfuscation technique but not necessarily malicious.

### Recommendation

Manual review recommended, but low confidence of actual maliciousness.
```
